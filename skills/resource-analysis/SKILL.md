# 工作流设计的资料分析

提供从用户提供的参考资料中提取工作流设计要素的方法论。

## 用途

从各类资源中提取可操作的工作流���计要素：
- API 文档
- 需求文档
- 流程描述
- 现有代码/脚本
- 数据库 Schema
- 业务规则

## 要素提取框架

### 目标要素

| 要素类型 | 关注点 | 用于 |
|---------|--------|------|
| **实体** | 名词、数据对象、资源 | Contract 设计 |
| **操作** | 动词、运算、转换 | Node 识别 |
| **流程** | 顺序、依赖、条件 | Flow 设计 |
| **数据结构** | 字段、类型、约束 | Schema 定义 |
| **规则** | 校验、业务逻辑、约束 | Validator 设计 |

### 提取流程

```
1. 识别资料类型
   ├── API 文档 → 关注端点、请求/响应 Schema
   ├── 需求文档 → 关注用户故事、验收标准
   ├── 流程描述 → 关注步骤、决策点、参与者
   ├── 代码/脚本 → 关注函数、数据转换
   └── 数据库 Schema → 关注表、关系、约束

2. 提取原始要素
   ├── 列出所有提及的实体
   ├── 列出所有操作/行为
   ├── 识别数据流
   └── 记录约束和规则

3. 映射到工作流概念
   ├── 实体 → 潜在的 Contract
   ├── 操作 → 潜在的 Node
   ├── 流程 → 执行顺序和条件
   └── 规则 → 校验逻辑

4. 组织输出
   ├── 分组相关要素
   ├── 识别依赖关系
   └── 标注需用户确认的歧义点
```

## 资料类型分析

### API 文档

**需识别的关键模式：**

| 模式 | 工作流要素 | 示例 |
|------|-----------|------|
| Endpoint | 潜在节点 | `POST /reviews` → create-review 节点 |
| 请求体 | 输入 Contract | JSON Schema → Contract Schema |
| 响应体 | 输出 Contract | JSON Schema → Contract Schema |
| 状态码 | 条件分支 | `200 OK`, `400 Bad Request` → ?success, ?error |
| 认证 | Context 需求 | API key → 环境变量 |

**提取模板：**

```markdown
## 来自 API: [Endpoint 名称]

### 识别的实体
- 名称: [实体名称]
- 操作: [可用的 CRUD 操作]

### 潜在节点
- 名称: [建议的节点名]
- 输入: [请求结构]
- 输出: [响应结构]

### 约束
- [来自 API 规范的校验规则]
```

### 需求文档

**需识别的关键模式：**

| 模式 | 工作流要素 | 示例 |
|------|-----------|------|
| "用户可以..." | 节点触发 | 用户可以提交评审 → submission 节点 |
| "系统应该..." | 节点动作 | 系统校验 → validation 节点 |
| "当...则..." | 条件流程 | 当有效时，继续 → ?valid 分支 |
| "必须有..." | 必填字段 | 必须有评分 → Schema 中的 required |
| "之前...之后..." | 顺序依赖 | 发布之前需评审 → a >> b |

**提取模板：**

```markdown
## 来自需求: [需求 ID/名称]

### 用户故事
[原始用户故事文本]

### 提取的工作流步骤
1. [故事中的步骤] → [建议的节点]
2. [故事中的步骤] → [建议的节点]

### 数据需求
- [必填字段]: [类型] - [约束]

### 业务规则
- [规则描述] → [校验逻辑]
```

### 流程描述

**需识别的关键模式：**

| 模式 | 工作流要素 | 示例 |
|------|-----------|------|
| 顺序步骤 | 流程顺序 | 步骤1, 步骤2 → a >> b |
| 决策点 | 条件分支 | 如果批准 → ?approved |
| 并行活动 | 并行组 | 同时 → [a, b] |
| 循环 | 迭代 | 对每个项目 → * $items |
| 参与者 | 节点职责 | 审核员检查 → reviewer 节点 |

**提取模板：**

```markdown
## 来自流程: [流程名称]

### 流程图
```
[原始流程描述或图表]
```

### 建议的 Flow DSL
```yaml
flow: |
  START >> [提取的流程] >> END
```

### 识别的角色/参与者
- [参与者] → [节点职责]

### 决策点
- [决策] → [条件表达式]
```

### 现有代码/脚本

**需识别的关键模式：**

| 模式 | 工作流要素 | 示例 |
|------|-----------|------|
| 函数 | 潜在节点 | `def process_data()` → processor 节点 |
| 输入参数 | 输入 Contract | 函数参数 → Schema 字段 |
| 返回值 | 输出 Contract | 返回类型 → Schema |
| 条件语句 | 分支条件 | `if status == 'ok'` → ?ok |
| 循环 | 迭代 | `for item in items` → * $items |
| 错误处理 | 错误分支 | `try/except` → ?error |

**提取模板：**

```markdown
## 来自代码: [文件/函数名称]

### 函数分析
- 名称: [函数名]
- 用途: [功能描述]
- 输入: [参数及类型]
- 输出: [返回类型]

### 建议的节点
```yaml
name: [建议的名称]
input:
  contract: [从参数推导]
output:
  contract: [从返回值推导]
```

### 需保留的逻辑
- [应放入 prompt 或 validator 的关键业务逻辑]
```

## 最佳实践

### 1. 全面但有组织

提取所有相关要素，但要组织清晰：
- 按类型分组（实体、操作、流程、数据、规则）
- 展示要素间的关系
- 标注依赖关系

### 2. 保留原始上下文

提取时：
- 引用相关原文
- 标注来源位置
- 记录所做的假设

### 3. 明确标注歧义

显式记录：
- 缺失的信息
- 不明确的需求
- 多种可能的解释
- 所做的假设

### 4. 建议而非决定

以建议的方式呈现：
- "建议的节点名: ..."
- "这可以映射到..."
- "考虑是否..."

### 5. 聚焦工作流相关要素

优先提取直接映射到以下概念的要素：
- Contract（数据结构）
- Node（执行单元）
- Flow（执行顺序）
- Context（环境需求）
