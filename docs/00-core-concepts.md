# 核心概念

cc-wf 的四个基本要素。所有上层设计（工作流编排、节点数据流、hook 机制等）都建立在这四个概念之上。

---

## Agent — 谁来干

**人设与身份定义。**

Agent 是执行者的角色抽象。它不关心具体干什么（那是 Command 的事），也不关心怎么干（那是 Skill 的事）。Agent 定义的是：

- **身份**：我是谁，我的职责边界是什么
- **行为倾向**：面对任务时的思维方式和决策偏好
- **能力声明**：我具备哪些 Skill（可组合）

一个 Agent 可以接受不同的 Command，用不同的 Skill 组合来完成它们。

**Claude Code 映射**：Subagent（`.md` agent 文件）

---

## Skill — 会什么

**可复用的能力抽象。**

Skill 是独立于具体任务的知识和方法论封装。它回答四个问题：

1. **做什么** — 这个能力解决什么类型的问题
2. **怎么做** — 方法论、流程、最佳实践
3. **用什么工具** — 需要哪些工具支持
4. **怎么用工具** — 工具的使用模式和约束

Skill 的关键特性：

- **可复用**：不绑定特定 Agent 或 Command，任何需要的地方都可以引用
- **可组合**：一个 Agent 可以同时装备多个 Skill
- **声明式**：Agent 声明需要哪些 Skill，运行时自动加载

**Claude Code 映射**：Skill（`SKILL.md` + 参考资料目录）

---

## Command — 干什么

**一次性的命令或任务。**

Command 是对"要做的事"的描述。它可以：

- 在任意时间节点发起
- 分派给任意 Agent 执行
- 携带参数来定制执行行为

Command 不关心谁来做（Agent 负责），不关心怎么做（Skill 负责），只关心做什么、要什么结果。

当 Command 编排多个 Agent 协作时，它就成了工作流的入口和调度器。

**Claude Code 映射**：Command（`.md` command 文件）

---

## 契约 — 做成什么样

**约束与规范检查。**

契约定义了两件事：

1. **规范** — 输入输出应该长什么样（结构、格式、必要字段）
2. **检查** — 如何验证结果是否符合规范

契约不参与执行过程，它只在关键节点做守门员：

- 数据进入 Agent 前 — 输入是否合法
- 数据离开 Agent 后 — 输出是否达标
- Agent 之间交接时 — 数据格式是否兼容

**Claude Code 映射**：Contract Schema（`.yaml`）+ Hook 校验脚本

---

## 四要素的关系

```
                    Command (任务)
                       │
                       │ 分派给
                       ▼
                    Agent (执行者)
                    │         │
          装备 Skill │         │ 遵守契约
                    ▼         ▼
            Skill (能力)   契约 (约束)
```

一句话总结：

> **Agent**（worker）带着 **Skill**（知识和工具）去执行 **Command**（任务），产出要符合**契约**（验收标准）。

---

## 与工作流的关系

这四个概念是原子级的，工作流是它们的编排：

| 单任务 | 工作流 |
|--------|--------|
| 1 个 Command 分派给 1 个 Agent | 1 个 Command 编排多个 Agent |
| Agent 用 Skill 执行，契约验收 | 每个 Agent 独立执行，契约在节点间守门 |
| 无状态传递 | 通过上下文（文件系统）传递状态 |

工作流不是第五个概念 — 它是 Command 的一种特殊形态：当一个 Command 需要多步骤、多 Agent 协作时，它自然演化为工作流。
